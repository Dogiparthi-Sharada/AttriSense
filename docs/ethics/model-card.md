# Model card

> What this model is, what it isn't, and the obligations that come with using it.

## Intended use

AttriSense is a **synthetic-data workforce intelligence demo**. It predicts voluntary-turnover risk, explains high-risk scores with SHAP, simulates possible retention actions via a Causal Uplift T-learner, and surfaces fairness audits at the department level.

The model is intended for **portfolio demonstration, analytics prototyping, and responsible-AI discussion**. It is **not approved for real employment decisions**.

## Data

The included records are 100% synthetic, generated by [`generate_demo_workforce_data.py`](https://github.com/Dogiparthi-Sharada/AttriSense/blob/main/generate_demo_workforce_data.py) with `RANDOM_SEED=42`. Each record carries:

- `employee_id`
- `department` (Engineering / Manufacturing / Sales)
- `tenure_months`
- `base_salary`
- `manager_id`
- `manager_tenure_months`
- `voluntary_turnover` (binary label)
- `exit_notes` (synthetic text)

Synthetic data keeps the repository public and safe, but it does not represent any real company's workforce distribution. The Manufacturing department is intentionally seeded with a stronger turnover signal so the [Fairness Audit](../features/fairness-audit.md) has a real disparate-impact failure to detect.

## Model

| | |
|---|---|
| Primary | Random Forest classifier (`n_estimators=300`, `max_depth=8`) |
| Class balancing | SMOTE on training fold only |
| Explainability | SHAP TreeExplainer for all high-risk + risk-weighted sample |
| Survival | Cox proportional hazards (lifelines) |
| Calibration | 10 reliability bins + Brier score |
| Causal uplift | EconML T-learner over 3 simulated treatment arms |

Full pipeline: [ML pipeline](../architecture/ml-pipeline.md).

## Performance (synthetic dataset)

| Metric | Value |
|---|---|
| Test AUC | ~0.85 |
| Brier score | ~0.10 |
| Calibration | well-calibrated below 0.7, slight over-confidence above |
| High-risk count | ~610 / 5,000 (12%) |
| Disparate-impact ratio (Engineering vs. Manufacturing) | 0.20 — **fails** four-fifths rule by design |

## Limitations

- Predictions are **correlational, not causal**. The Causal Uplift tab uses simulated treatment assignment — its uplift estimates are demoware.
- Synthetic labels encode simplified risk patterns and should not be treated as real workforce behaviour.
- Department and manager signals must be audited before any production deployment.
- The Cox survival model is a demo time-to-event layer and requires real longitudinal validation before use.
- The fallback NL→SQL ranker uses TF-IDF — it cannot answer novel questions, only rank against the 50 hand-validated gold questions.

## Responsible use policy

AttriSense should be used to **support employees, not punish them**. Scores must never be used as the **sole basis** for:

- Termination
- Compensation denial
- Promotion denial
- Disciplinary action

Any real deployment must include:

- **Independent bias audit** by a third party.
- **Human review** before any action.
- **Clear notices** to employees and candidates where required by law.
- **NYC Local Law 144** review if used in covered employment-decision contexts.
- **Documented thresholds, monitoring, and appeal process**.

See [Intended Use](intended-use.md) for the full boundary.

## Monitoring checklist

- [ ] Track calibration drift quarterly (`outputs/metrics_snapshot.txt`)
- [ ] Review risk rates by department and manager group monthly (built-in)
- [ ] Compare suggested interventions against actual retention outcomes (requires retrospective data)
- [ ] Revalidate SHAP driver stability after each retrain
- [ ] Maintain an audit log for any HR action influenced by model output
- [ ] Run NL→SQL eval harness on every dependency upgrade

## Out of scope

- Hiring decisions
- Compensation benchmarking
- Performance reviews
- Background-check signal generation
- Any use case that would benefit a third party at the employee's expense

If you are tempted to extend AttriSense into any of these, **stop and re-read this section**.
